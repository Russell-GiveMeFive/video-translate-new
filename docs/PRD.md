## 1. 背景与商业语境

### 1.1 趋势

2025–2026 年中国 AI 短剧呈爆发式增长，单部成本从传统真人短剧的数十万降至千元级，**单位内容产能放大 100–1000 倍**。当国内分账市场进入存量竞争，「**短剧出海**」（ReelShort / DramaBox / ShortMax 等海外平台）成为下一个核心增量战场——东南亚、北美、拉美、中东都有显著付费意愿。

### 1.2 痛点

出海的第一道工程化关卡就是「**语言**」：

| 维度 | 现状 | 痛点 |
| :-- | :-- | :-- |
| 翻译 | 人工字幕组、机翻 | 慢、贵、口语化差 |
| 配音 | 外包真人录音 | 单集 ¥3,000+、周期 3-7 天、说话人音色和原片差距大 |
| 同步 | 人工对轨、剪辑 | 时长不匹配导致口型 / 情绪错位 |
| 字幕 | 手工 OCR / 重打 | 烧字幕的视频极难干净抠除 |

### 1.3 机会

随着 MiniMax Speech-2.8 / M3、火山豆包 Seed-ASR、各家 Voice Clone 在 2025 年集中跃迁，**端到端「中文短剧 → 任意语种译制片」可在 30 分钟内、单集成本 < ¥30 完成首版**——这就是本产品的 raison d'être。

## 2. 目标用户与场景

### 2.1 主要用户画像

| 画像 | 角色 | 关心什么 |
| :-- | :-- | :-- |
| **P1 短剧厂牌制片** | 在「点众 / 九州 / 容量 / 麦芽」等公司的译制 PM | 一键出片、批量、节省外包成本、合规可审计 |
| **P2 独立译制工作室** | 5-20 人小团队，承接出海订单 | 译制质量稳定可控、可人工微调（半成品交付） |
| **P3 个人创作者 / MCN** | 抖音/快手切片号、海外搬运 | 用得起（按集付费）、操作门槛低、不需要懂剪辑 |

### 2.2 典型使用场景

1. **批量首版**：制片把 80 集 × 90 秒短剧拖入产品，挂着过夜，第二天拿到 80 集英 / 西 / 印尼三语版本。
2. **单集精修**：译制师拿到首版后，进入「编辑模式」逐句调整翻译、替换音色、微调时长。
3. **音色资产管理**：长剧集需要主角音色稳定，将主角声音永久化为「我的音色库」复用。
4. **样片试播**：销售拿首集 10 分钟做 demo 给海外平台，**5 分钟内产出**比拼质量。

### 2.3 非目标用户（暂不服务）

- 真人电影 / 电视剧（高级口型对齐、版权链路复杂）❌
- 综艺 / 直播 / 多人长场景（说话人分离 > 8 人）❌
- 实时直播翻译 ❌

## 3. 产品定位与范围

### 3.1 一句话定位

> **「短剧译制工作站」**——桌面级、端到端、可批量、可精修的 AI 译制软件。

### 3.2 In-Scope（v1.0 MVP）

- 单集 / 批量视频导入（mp4 / mov / mkv，单文件 ≤ 2 GB）
- **导入前素材准入检查**：自动检测视频是否含硬字幕，若有则警告并引导用户准备无字幕原片（v1.0 不做本地硬字幕抠除）
- 自动 ASR + 自动说话人分离（diarization）
- 字幕 OCR（用于翻译辅助 + 译文长度参考；不做抠除）
- LLM 翻译（中 → **全 40 语种**，与 MiniMax Speech-2.8 支持范围对齐，详见附录 §A 语种工程矩阵）
- 说话人音色克隆（按角色）
- TTS 合成 + 时长对齐
- 视频合成回填（音轨替换 + 可选新字幕烧录）
- 项目管理 / 工程文件保存 / 中间产物可复用
- 桌面端 Win + Mac 双平台

### 3.3 Out-of-Scope（v1.0 不做）

- ❌ **硬字幕抠除（subtitle inpaint）**——v1.0 改为「**素材准入检查 + 用户素材准备指南**」策略；v1.1 评估**云端 GPU 抠除服务**（D4 ✅）
- ❌ 高级**口型同步**（Wav2Lip 类）—— 列入 v2.0
- ❌ **背景音乐 / SFX 重建**（仅做人声/背景分离后保留背景轨）
- ❌ 真人**画面替换**（如人脸翻译/重渲染）
- ❌ **云端协作** / 多人编辑（v1.0 单机）
- ❌ **移动端**

> 🚨 **D4 风险声明**：国内短剧 99% 是硬字幕烧录成片。v1.0 不做本地抠除意味着**用户必须自行准备无硬字幕的原始素材**（如向制片方索取原片、或先用第三方工具抠除）。产品在 **导入阶段** 通过硬字幕自动检测 + 醒目警告 + 素材准备指南文档来降低误用风险，但**带硬字幕成片直接走完译制流程会产生「中英字幕叠加」的不可用结果**，需在 onboarding 与官网文档中**反复强调**。

### 3.4 设计原则

1. **「半成品 + 编辑器」优于「黑盒一键」**：用户必须能够干预每个环节，错了能修，不能全部重跑。
2. **「断点续跑」是底线**：任何阶段失败/被打断，都能从上一个 checkpoint 恢复，不重复消费 token。
3. **「外部 API 可替换」**：所有云端服务通过 Provider 抽象层，便于未来增加 OpenAI / Azure / 自研模型。
4. **「成本可见」**：每一步都展示预估 / 已花费的 token / 时长 / ¥，决不让用户「不知不觉烧钱」。

## 4. 核心概念与术语

| 术语 | 含义 |
| :-- | :-- |
| **项目（Project）** | 一次译制任务的工程文件，含源视频、目标语种、所有中间产物与编辑结果 |
| **场景（Scene）** | 视频按 shot detection 切分后的镜头段 |
| **片段（Segment / Cue）** | 按时间轴切分的一句台词（句级），与 ASR 输出对应 |
| **角色（Character）** | 同一说话人的归类，绑定一个克隆音色 |
| **音色（Voice）** | MiniMax `voice_id`；分**系统音色 / 临时复刻音色（7 天）/ 永久复刻音色** |
| **时间轴（Track）** | 视频轨 / 原始人声轨 / 背景音轨 / 译制人声轨 / 字幕轨 |
| **对齐策略（Fit Strategy）** | 处理译文音频与原段时长不匹配的算法（见 §9） |
| **Provider** | 外部 API 提供商抽象（MiniMax / 火山豆包 / …） |

## 5. 端到端业务流程

```
┌─────────────┐
│  导入视频    │ ① 用户拖入 mp4
└──────┬──────┘
       ▼
┌─────────────┐
│ 预处理       │ ② ffprobe 抽元数据，shot detection，缩略图
└──────┬──────┘
       ▼
┌─────────────┐
│ 音轨分离     │ ③ 人声 / 背景（Demucs 本地 or 云端 stem split）
└──────┬──────┘
       ▼
┌─────────────┐
│ ASR + 分轨   │ ④ 豆包 Seed-ASR → 时间戳 + 说话人 ID
└──────┬──────┘
       ▼
┌─────────────┐
│ 字幕 OCR     │ ⑤ OCR 文本辅助 ASR 校验 + 译文长度参考（不做抠除）
└──────┬──────┘
       ▼
┌─────────────┐
│ 角色聚类     │ ⑥ 说话人 embedding 聚类 → 角色列表，给用户命名
└──────┬──────┘
       ▼
┌─────────────┐
│ 音色复刻     │ ⑦ 每个角色挑选 10-30s 干净人声 → MiniMax voice clone
└──────┬──────┘
       ▼
┌─────────────┐
│ LLM 翻译     │ ⑧ M3 整段 + 上下文翻译，保口语 / 性别 / 情绪
└──────┬──────┘
       ▼
┌─────────────┐
│ 用户审校     │ ⑨ （可选）句级编辑器调整译文 / 角色 / 时长策略
└──────┬──────┘
       ▼
┌─────────────┐
│ TTS 合成     │ ⑩ Speech-2.8 + 复刻音色，按句生成音频
└──────┬──────┘
       ▼
┌─────────────┐
│ 时长对齐     │ ⑪ 变速 / 重生成 / 微间隙吸收 → 与原段对齐
└──────┬──────┘
       ▼
┌─────────────┐
│ 字幕烧录     │ ⑫ 烧入新字幕（可选；硬字幕抠除 v1.0 不做，见 D4）
└──────┬──────┘
       ▼
┌─────────────┐
│ 视频合成     │ ⑬ ffmpeg 合并：原视频 + 译制人声 + 背景 + 新字幕
└──────┬──────┘
       ▼
┌─────────────┐
│ 导出 / 复用  │ ⑭ 导出 mp4 / srt / 工程包；音色入库
└─────────────┘
```

每个阶段都是独立的 Pipeline Stage，状态持久化到本地 SQLite，**支持断点续跑**。

## 6. 功能需求（FR）

### 6.1 项目管理（FR-PRJ）

| ID | 描述 |
| :-- | :-- |
| FR-PRJ-01 | 新建项目：选择源视频、源语言（默认中文）、目标语言、目标平台规格（如 ReelShort 9:16 1080P） |
| FR-PRJ-02 | 项目列表 / 搜索 / 排序（按创建时间、状态、目标语种） |
| FR-PRJ-03 | 项目工程包导出 / 导入（`.dpx` zip 含元数据 + manifest + 中间产物引用） |
| FR-PRJ-04 | 项目级配置：覆盖全局默认（如本项目使用哪个 LLM 模型 / TTS 模型） |
| FR-PRJ-05 | 项目克隆：基于已有项目克隆配置和音色，仅替换源视频 |
| FR-PRJ-06 | 项目级成本面板：累计 token / 时长 / ¥ |

### 6.2 批量任务（FR-BATCH）

| ID | 描述 |
| :-- | :-- |
| FR-BATCH-01 | 拖入文件夹 / 多文件 → 自动逐个建项目 |
| FR-BATCH-02 | 批量模板：一份配置（目标语言、角色音色映射、对齐策略）应用到 N 集 |
| FR-BATCH-03 | 队列调度：可控并发数（默认 2），失败自动重试 3 次，错误归类 |
| FR-BATCH-04 | 批处理完成后桌面通知 + 报告（成功 / 失败 / 部分） |
| FR-BATCH-05 | 「跨集音色继承」：1-N 集共用同一组角色音色（同剧情况） |

### 6.3 预处理（FR-PRE）

| ID | 描述 |
| :-- | :-- |
| FR-PRE-01 | 自动检测视频分辨率 / 帧率 / 码率 / 音轨数 / 时长，展示元数据 |
| FR-PRE-02 | 缩略图与波形可视化 |
| FR-PRE-03 | 镜头切分（PySceneDetect / 阈值可调）便于后续翻译时考虑场景上下文 |
| FR-PRE-04 | 人声 / 背景音轨分离（默认本地 Demucs；可切换云端） |

### 6.4 ASR + 说话人分离（FR-ASR）

| ID | 描述 |
| :-- | :-- |
| FR-ASR-01 | 调用火山豆包大模型 ASR（含时间戳、标点、说话人 ID） |
| FR-ASR-02 | ASR 结果可视化：字幕轨 + 说话人色条 |
| FR-ASR-03 | 句级 ASR 文本可编辑（修正同音字、识别错误） |
| FR-ASR-04 | 说话人聚类阈值可调（默认 0.7） |
| FR-ASR-05 | 失败兜底：本地 Whisper-large 作为备份引擎（v1.1） |

### 6.5 OCR & 字幕处理（FR-OCR）

> **D4 ✅ 范围调整**：v1.0 不做本地硬字幕抠除（inpaint）。OCR 仅用于「**辅助 ASR 校验** + **译文长度参考**」；硬字幕检测仅用于**导入前的素材准入警告**。

| ID | 描述 |
| :-- | :-- |
| FR-OCR-01 | **导入预检（preflight）**：扫描首/中/尾 30 帧，检测视频是否含硬字幕；如含，弹窗警告 + 引导文档链接「如何准备无字幕原片」；用户可选「忽略并继续」（结果可能不可用） |
| FR-OCR-02 | OCR 识别原视频字幕文本（如有），与 ASR 文本做交叉校验（提高准确率 5-10%）；用于**译文长度参考**（参照原字幕换行节奏） |
| FR-OCR-03 | ~~硬字幕抠除~~ ❌ **v1.0 不做**（参见 §3.3 D4 风险声明） |
| FR-OCR-04 | 新字幕样式编辑：字体、字号、位置、描边、阴影；**双语对照仅支持中-英**配对（D13），其它语种为单语字幕（RTL 三语正确单语渲染） |
| FR-OCR-05 | 字幕导出：SRT / VTT / ASS |
| FR-OCR-06 | 新字幕位置避让：默认放置在视频底部 1/8 区域；若检测到原硬字幕也在该区域，自动改放到底部 2/8 区域上方，**减少视觉冲突**（无法消除，仅缓解） |

### 6.6 角色与音色（FR-CHAR）

| ID | 描述 |
| :-- | :-- |
| FR-CHAR-01 | 自动聚类后展示角色列表（默认命名「角色 1/2/3」+ 头像缩略 + 试听片段） |
| FR-CHAR-02 | 用户可手动重命名、合并、拆分角色 |
| FR-CHAR-03 | **全自动**挑选每个角色的克隆样本（D10 ✅）：综合评分 = SNR×0.4 + 时长得分×0.3 + 背景音乐反向得分×0.3，自动选最佳一段；用户在编辑器可手动重选 |
| FR-CHAR-04 | 一键克隆：调 MiniMax `voice_cloning_clone` → 拿到临时 `voice_id` |
| FR-CHAR-05 | 音色试听：用 TTS 合成一句样本，确认音色匹配 |
| FR-CHAR-06 | 音色重新克隆 / 替换为系统音色 / **手动重选样本片段**（FR-CHAR-03 的兜底入口）|
| FR-CHAR-07 | **音色资产库**：将临时音色「永久化」（首次合成触发计费）、命名、标签、跨项目复用 |
| FR-CHAR-08 | 音色到期提醒（< 24h 推送，避免临时音色失效） |

### 6.7 翻译（FR-TRANS）

| ID | 描述 |
| :-- | :-- |
| FR-TRANS-01 | 默认用 MiniMax M3，整段（场景级）批量翻译，prompt 注入角色性别 / 情绪 / 场景上下文 |
| FR-TRANS-02 | 译文与原文逐句双栏对照 |
| FR-TRANS-03 | 译文支持「重译此句」「重译此场景」「整集重译」 |
| FR-TRANS-04 | 术语表 / 译名表（人名、地名、专有词）项目级 + 全局 |
| FR-TRANS-05 | 译文长度自动控制：在 prompt 中传入「目标字符数 / 音节数预算」，使 TTS 时长偏差 < 15% |
| FR-TRANS-06 | 译文风格选择：直译 / 本地化 / 字幕化（短促）/ 配音化（口语） |
| FR-TRANS-07 | 不当词 / 敏感词高亮提示（按目标地区合规库） |

### 6.8 TTS 合成（FR-TTS）

| ID | 描述 |
| :-- | :-- |
| FR-TTS-01 | 调用 MiniMax Speech-2.8-hd / turbo，按句合成 PCM/WAV |

### 8.2 桌面框架（已决议 D1 ✅）

**Electron 30+ + Node 20 (TypeScript) + React 18 + Vite + Tailwind + shadcn/ui**。

> Electron 选型后必须立刻规避的 4 个已知坑（写入工程红线）：
>
> 1. **大文件不进 IPC**：源视频 / 中间音频禁止经 `ipcRenderer.invoke` 传二进制，全部通过**文件路径引用**，Renderer 仅持有 path + 元数据。
> 2. **重任务进 utilityProcess**：转码、Demucs、ffmpeg、上传等阻塞操作放进 `utilityProcess.fork`，避免阻塞 Main，避免 Renderer 卡死。
> 3. **安全基线**：`nodeIntegration: false` / `contextIsolation: true` / `sandbox: true`；所有 Renderer → Main 通道通过 `contextBridge` 白名单暴露。
> 4. **安装包瘦身**：使用 `electron-builder` + `asar` + 按需下载模型 / 字体 / Python sidecar；目标 ≤ 300MB 见 §7 NFR。

### 8.3 模块划分

| 模块 | 职责 | 实现 |
| :-- | :-- | :-- |
| `core-pipeline` | 编排 14 个阶段，状态机驱动 | Node TS（Main） |
| `core-storage` | SQLite（项目元数据）+ 文件系统（产物） | `better-sqlite3` |
| `core-ffmpeg` | 视频/音频读写、转码、合成 | spawn ffmpeg CLI（静态二进制随包） |
| `core-providers` | MiniMax / 火山豆包 HTTP/WS 客户端 | Node `undici` + `ws` |
| `core-keystore` | API Key 加密存取 | `keytar`（macOS Keychain / Windows DPAPI） |
| `core-workers` | 重活进 utilityProcess / worker_threads | Node `utilityProcess` |
| `py-runner` | Demucs / Diarization / OCR / Inpaint Python 子进程 | Python 3.11（PyInstaller `--onedir`） |
| `ui` | 全部界面 | React 18 + TypeScript + Tailwind + shadcn/ui + Vite |

### 8.4 数据流（一次单集译制）

1. Renderer 触发 `window.api.startProject(projectId)` → contextBridge → Main
2. Main `PipelineOrchestrator` 加载项目，构造 DAG，按 stage 顺序调度
3. 每个 stage：
   - I/O-heavy 阶段（ASR / TTS / LLM）→ Node HTTP 客户端（`undici`，连接池 + keepalive）
   - CPU/GPU-heavy 阶段（Demucs / OCR / Inpaint）→ 调 `py-runner` 子进程，stdin/stdout JSON-RPC
   - FFmpeg / 长上传 → `utilityProcess.fork` 隔离，避免阻塞 Main
4. stage 结束写回 SQLite + 中间文件，通过 `webContents.send` 推进度事件给 Renderer
5. 失败：标记 stage = `failed`，UI 可重试该 stage（前置产物复用）

### 8.5 Python Sidecar 打包

- 用 `pyinstaller --onedir` 把 Python 解释器 + 依赖打成独立目录
- macOS 用 universal2，Windows 分别打 x64
- 模型权重首次启动按需下载到 `~/.dramaprime/models/`（不打进安装包）

## 9. 关键算法策略

### 9.1 时长对齐（最重要！）

**问题**：原句中文 3.2s「你今天怎么这样对我」→ 英文 TTS「Why are you treating me like this today?」≈ 4.1s，差 0.9s。

**多策略级联**：

```
┌─────────────────────────────────────────────────────┐
│ Stage 1: 控长翻译                                    │
│  - prompt 注入 target_char_budget = round(orig*K)   │
│  - K 为语种对系数（zh→en ≈ 1.6, zh→es ≈ 1.8）       │
│  - 失败：translation too long → 触发 Stage 2        │
└─────────────────────────────────────────────────────┘
                       │ 仍超时长
                       ▼
┌─────────────────────────────────────────────────────┐
│ Stage 2: TTS 语速调节                                │
│  - 计算 ratio = target_dur / synth_dur              │
│  - 在 [0.85, 1.15] 内调 speed 参数重合成            │
│  - 超出区间 → 触发 Stage 3                          │
└─────────────────────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────┐
│ Stage 3: 弹性变速（SOLA / WSOLA / Rubber Band）     │
│  - 不变音高的时间拉伸，可在 ±30% 内保持自然          │
│  - 应用于已合成 PCM                                  │
└─────────────────────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────┐
│ Stage 4: 间隙吸收 / 借用                             │
│  - 若与下一句之间存在 silence > 200ms，可向后延展    │
│  - 或借用前一句的尾静音                              │
└─────────────────────────────────────────────────────┘
                       │ 极端情况
                       ▼
┌─────────────────────────────────────────────────────┐
│ Stage 5: 视频局部慢放（可关闭）                       │
│  - 仅在该镜头无强动作时启用                          │
│  - ±10% 内通常人眼难察觉                            │
└─────────────────────────────────────────────────────┘
```

❓ **决策点 D2**：是否默认启用 Stage 5（视频局部变速）？这会影响输出片源「物理保真度」。

### 9.2 角色聚类与音色复刻

- ASR 自带 diarization → 得到说话人段
- 提取每段说话人 embedding（ECAPA / 3D-Speaker）
- 聚类（HDBSCAN，按 embedding cosine）→ 角色组
- 每个角色挑选：**单段时长 ≥ 10s、SNR > 20dB、无背景音乐**
- 调 MiniMax `voice_cloning_uploadcloneaudio` → `voice_cloning_clone`
- 返回 `voice_id`，标记**临时**（168h）

❓ **决策点 D3**：角色 < 5s 时怎么办？
- A. 跳过克隆，用系统音色匹配（按性别 / 年龄推荐）
- B. 拼接同人多段达到 10s
- C. 让用户决定

### 9.3 字幕去除（Inpaint） ❌ v1.0 不做（D4 ✅）

v1.0 不实现本地硬字幕抠除。替代策略详见 §3.3 风险声明与 §6.5 FR-OCR。**v1.1 评估方案**：

- **方向 A（推荐评估）**：接入云端 GPU 抠除服务（自建 RunPod / 火山 GPU 弹性实例 + ProPainter / LaMa），按集计费转嫁给用户
- **方向 B**：开放本地 GPU 抠除作为可选高级功能，要求 NVIDIA RTX 30+ 或 Apple Silicon M2+
- **方向 C**：与第三方"字幕抠除工具"集成，由用户在产品外预处理

决策时机：v1.0 GA 后收集用户对该功能的明确诉求量再决定优先级。

### 9.4 翻译 Prompt 设计（关键）

`★ Insight ─────────────────────────────────────`
LLM 翻译短剧的瓶颈不是「翻得对」，而是「翻得像配音」。短剧台词高度口语化、情绪强烈，直译会显得呆板。Prompt 必须包含：场景描述、角色性别 / 年龄 / 情绪、目标地区文化偏好、字数预算。
`─────────────────────────────────────────────────`

模板（伪代码）：

```
SYSTEM: 你是短剧译制专家，将中文台词翻译为 {target_lang}。
**地区中性化（D11 ✅）**：UI 不暴露地区变体，但内部 prompt 必须按附录 A.2「地区中性化要点」执行——
例如 es 默认 LATAM 中性、pt 默认 pt-BR、ar 走 MSA、fr 默认欧法、nl 标准荷语。规则库见 `prompts/region_neutral/{lang}.yaml`。
要求：
1. 口语化、符合 {target_lang} 配音节奏
2. 保留情绪强度（愤怒/撒娇/冷淡），不弱化
3. 角色性别 / 称谓符合 {character.gender}, {character.age}
4. 每句长度 ≤ {char_budget} 字符
5. 若原句使用中式比喻，转为 {region} 受众熟悉的等价比喻

USER:
[场景上下文：豪门家庭客厅，男女主角对峙]
[前 3 句历史] ...
[本场全部台词，含 speaker_id + 原文 + 时长]
{
  "1": {"speaker":"A_female_25","text":"你今天怎么这样对我","dur":3.2},
  ...
}

返回 JSON：
{"1":{"text":"...", "est_dur":3.4}, ...}
```

**D5 ✅ 决议**：**不做整集二次润色**。一次性逐场翻译完成。理由：
- 单集 LLM 成本控制在 ¥0.08 内（润色后翻倍至 ¥0.16，批量场景成本敏感）
- 短剧台词稳定性高（场景切换明确，称谓上下文窄）
- 一致性问题由「术语表 / 译名表」+「跨场景 prompt 注入前 N 句历史」覆盖
- 后期若推出"精修模式"付费档，可作为增值功能引入

### 9.5 ASR 准确性增强

- ASR 文本 + OCR 字幕文本对齐（DTW），冲突时优先 OCR（短剧字幕基本与台词逐字一致）
- 用户级术语表注入 ASR / 翻译 prompt
- 角色台词风格学习（v2）：根据已翻译片段学个性化用语

## 16. 风险与缓解

| 风险 | 影响 | 缓解 |
| :-- | :-- | :-- |
| MiniMax / 火山 API 不稳 | 阻塞主流程 | Provider 抽象层；多次重试 + 备用通道（本地 Whisper） |
| QPS 限流 | 批量速度受限 | 客户端限流器 + 队列 + 优先级 |
| 音色 7 天到期 | 长期项目音色失效 | 到期前提醒，UI 一键永久化 |
| 时长对齐效果差 | 观感差 | 多策略级联 + 严苛模式人工兜底 |
| **用户拖入带硬字幕成片 → 双字幕叠加** | 不可用结果，用户流失 | 导入预检 + 醒目警告 + 素材准备指南；落地页 FAQ 明确「需准备无字幕原片」；v1.1 评估云端抠除 |
| 字幕烧录与原硬字幕视觉冲突 | 观感差 | FR-OCR-06 自动避让位置；用户可在编辑器自行调整 |
| 内容合规 | 法务风险 | 上传前不强制云端分发；用户自负内容；本地敏感词预检 |
| 安装包过大 | 下载流失 | 模型按需下载；首装 ≤ 300MB |
| API Key 泄漏 | 用户损失 | 系统 keystore；最小权限；日志脱敏；本地不持久化明文 |
| 用户输入版权片 | 法务 | 上传前免责声明；不主动上传到非签约服务 |

### A.3 RTL 工程清单（3 种语言：ar / he / fa）

**范围说明（D13 ✅）**：v1.0 仅做 **RTL 单语字幕** 的正确渲染；**不**实现「LTR 中文 + RTL 阿语」的同屏混排字幕。RTL 字幕在编辑器内部仍可看到「中文原文 + RTL 译文」上下分行的预览（用于译制师参考），但最终烧入视频的双语对照只在 **中-英** 配对上提供。

| 模块 | RTL 适配点 |
| :-- | :-- |
| 字幕渲染（单语 RTL） | `direction: rtl`、`text-align: right`、标点镜像（"，" → "،"）；ffmpeg `subtitles` filter 走 libass 原生 BiDi |
| 字幕编辑器（单语 RTL） | 输入框 `dir="rtl"`，光标方向；插入中文时按 BiDi 算法处理 |
| 时间轴 | 进度条方向不翻转（仍 L→R，避免操作混乱）；只翻转字幕预览区域 |
| 双语对照（仅中-英） | 中文 + 英文上下两行；同语种相同方向无 BiDi 复杂度 |
| 烧字幕 | ffmpeg `subtitles` filter 配 `force_style` + libass |
| **不做** | LTR + RTL 混排（如中阿同帧）；如未来 v2.0 需要，再做 BiDi 段落级算法 |

### A.4 字体打包策略

- **基础包（默认安装）**：Latin / CJK 子集（中英日韩） ≈ 20MB
- **扩展包（按需下载）**：Noto Sans Arabic / Hebrew / Devanagari / Thai / Tamil / Khmer / Greek / Cyrillic ≈ 60MB
- 首次选择新语种 → 自动检测是否缺字体 → 弹窗引导下载

| **校准协调人（内部）** | 1 名（跨语种共用） | 团队内部 | 负责派单、聚合、出报告 |

#### A.5.2 评测样本与流程

```
Step 1  样本采集
        - 从 5 集真实短剧抽取 200 句样本（覆盖对白/独白/旁白/情绪台词/俗语）
        - 抽样规则：句长分布 ≈ 真实分布；至少含 30% 高情绪台词

Step 2  对照集准备（P0 才做）
        - P0 5 语种：每语种购买 200 句人工翻译参考（¥3-5/句，外包译制公司）
        - P1 15 语种：跳过参考集，直接做主观盲测

Step 3  我方系统跑全流程
        产出：译文 + TTS 音频 + 时长偏差数据

Step 4  人工盲测（核心）
        每位评测员独立打分，不知道是机器还是人工译制（如有参考集则混入）：
        - 翻译准确性（1-5）
        - 翻译自然度（1-5，是否像母语短剧台词）
        - 情绪保留度（1-5）
        - 音色匹配度（1-5，与原片角色一致性）
        - 整体观感 MOS（1-5）
        - 不可接受问题打标：错译 / 漏译 / 语法错 / 文化不当 / 音色违和 / 时长突兀

Step 5  数据聚合与指标计算
        - BLEU vs 参考集（仅 P0）
        - 字数膨胀系数 K' = avg(译文字符数 / 原文字符数)
        - 时长系数 R' = avg(TTS 时长 / 原始时长)
        - 整体 MOS = 大众评测员 trim-mean（去掉最高最低各 1）
        - 不可接受问题率 = 标记数 / 总句数

Step 6  Lead Reviewer 复评
        首席评测员对所有「不可接受问题」逐条复核，输出该语种典型错误模式

Step 7  更新工程参数
        - 更新附录 A.2 表格的 K 与节奏系数
        - 追加该语种 prompt 加强规则（如「西班牙语 LATAM 避免使用 vosotros」入库）
        - 更新地区中性化要点列

Step 8  纳入回归集
        - 评测样本进入回归集；每次版本迭代必跑
        - 整体 MOS 不能比上版本降 0.2 分以上（否则阻断发版）
```

#### A.5.3 P0 / P1 / P2 准入与发版门槛

| 分级 | 准入要求（首次校准） | 发版门槛（每次迭代） |
| :-- | :-- | :-- |
| **P0** | 整体 MOS ≥ 4.0；不可接受问题率 ≤ 5% | MOS 不下降 > 0.1；问题率不上升 > 1pp |
| **P1** | 整体 MOS ≥ 3.8；不可接受问题率 ≤ 10% | MOS 不下降 > 0.2；问题率不上升 > 2pp |
| **P2** | 仅冒烟测试：3 集端到端无技术故障 | 冒烟通过即可 |
